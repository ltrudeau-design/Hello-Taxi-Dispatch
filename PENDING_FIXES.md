## 🚀 Future Architecture — Hello Taxi MCP Server

### Overview
Hello Taxi should eventually become an MCP server, making it a first-class Cluster citizen. The browser app becomes a display/control surface; the MCP server becomes the authoritative data and operations layer.

### Proposed MCP Tools
| Tool | Description |
|------|-------------|
| `create_call_card` | Create a new dispatch call (pickup, dropoff, client, notes) |
| `assign_driver` | Assign a driver to a queued call |
| `complete_trip` | Mark a trip as complete, log fare |
| `list_queue` | Return current dispatch queue |

| `add_client` | Add/update a client record |
| `add_place_nickname` | Add a place nickname → address mapping |
| `get_shift_report` | Return shift stats (trips, revenue, drivers) |
| `get_client_by_phone` | Look up client by phone number |
| `parse_dispatch_input` | LLM-parse raw text/dictation into a call card |

### Benefits
- EauDev can dispatch directly via voice → intent → MCP tool call
- Cluster can read/write dispatch data without browser interaction
- Cross-session persistence (proper database, not localStorage)
- Analyst MCP can pull shift reports directly for historical analysis
- Voice-first dispatch — browser becomes display layer only

### Migration Path
1. Extract `DATA` schema from localStorage to SQLite or JSON file
2. Wrap existing JS logic as Python functions in MCP server
3. Browser app calls MCP server via WebSocket (replacing direct localStorage reads)
4. EauDev gains direct dispatch capability
5. Cluster Archive MCP syncs client/nickname data automatically

## 🎯 Dedicated Session — Address Resolution Overhaul

### The Core Problem
The address pipeline is brittle because nickname resolution and geocoding are loosely coupled and inconsistent. Failures are silent.

### Current Flow (broken points marked)
```
Raw dictation
    → LLM parses → outputs address strings (may still contain nicknames) ⚠️
    → autoGeocodeAndEstimate()
    → geocodeAddressInternal()
    → Google Places TextSearch (wrong API for addresses) ⚠️
    → lat/lon or silent failure ⚠️
```

### Ideal Flow
```
Raw dictation (full telephone conversation)
    → LLM: extract name, phone, pickup, dropoff, waypoints, time
           resolve nicknames using injected context
           output clean full addresses
    → Nickname pre-resolution step (catch any LLM misses)
    → Google Geocoder API (not Places TextSearch)
    → lat/lon validated
    → Call card populated
```

### Key Changes Needed
1. **Nickname pre-resolution** — before any Google API call, check address against `DATA.placeNicknames`. If match found, substitute full address. Google always gets clean input.
2. **Switch from Places TextSearch to Geocoder API** — more reliable for specific addresses in small cities like Steinbach.
3. **Graceful failure handling** — if geocoding fails, flag the field visually rather than silently failing. Dispatcher can correct manually.
4. **LLM prompt refinement** — test dictation of full telephone conversations (not just address summaries). Verify LLM correctly extracts all fields from natural conversational text.

### Test Cases to Prepare
- Regular client by name only ("it's John Smith")
- Nickname only ("pickup at St. Joe's, dropoff the mall")
- Partial address ("45 Main" — missing city)
- Ambiguous dropoff ("downtown")
- Prebook with time ("tomorrow at 2:30, from 123 Elm to the airport")
- Multi-stop ("pickup 45 Main, stop at Walmart, dropoff 67 Oak")
- Return trip ("same address back")

### Session Prerequisites
- V105.1 stable and tested in production
- Full list of current `DATA.placeNicknames` entries reviewed
- Google Maps API key and quota confirmed
- Quiet shift — not a busy Friday night

**Deliverable:** A data flow map — every cross-kernel dependency documented. Inconsistencies flagged for fixing.

---

### Session 2 — Global ETA Refactor
**Goal:** Fix Global ETA to produce reliable, real-world estimates.
**Depends on:** Session 1 (data flow audit complete, inputs verified reliable)

**The correct formula:**
```
For each clocked-in driver:
    committed_time =
        active trip remaining (activeTrip.end - now, including ALL added minutes)
        + stacked trips duration (each with added minutes)
        + Google Routes: final drop-off location → city center

Soonest driver = min(committed_time across all drivers)
Global ETA = soonest driver's committed_time
```

**Key rules:**
- Formula is identical for idle, active, and stacked drivers
- Idle driver: last drop-off location → city center (one Google Routes call)
- Active driver: remaining time + drop-off → city center
- Stacked driver: remaining active + stacked durations + final drop-off → city center
- Minute additions per phase MUST flow through to activeTrip.end — verify this in Session 1
- No queue wait addition — queue depth already reflected in committed time via stacked trips
- No weather/temporal multipliers — they add false precision to a gross estimate
- No Haversine — replace with Google Routes for the drop-off → city center leg

**Known failure mode to avoid:**
Previous implementations produced nonsense (idle driver = 40 min ETA) because:
1. Queue wait was added on top of committed time (double counting)
2. Minute additions weren't updating `activeTrip.end` (stale inputs)
3. Haversine to city center used when driver position was unknown (defaulted to 10km)

**Deliverable:** Global ETA that produces reliable 5-30 minute estimates matching real-world dispatch experience.

## 🏷️ Trip Name Fix — Post-Refactor

### Bug
`generateSimplifiedTripName()` in `kernel-dispatch.js` line ~945 explicitly strips address numbers:
```javascript
var streetMatch = simplified.match(/^\d+\s+(.+)$/);
if (streetMatch) {
    simplified = streetMatch[1]; // ← strips number, keeps street name only
}
```
Result: "123 Main Street → McDonald's" becomes "Main St → McDonald's"

### Fix
```javascript
var streetMatch = simplified.match(/^(\d+)\s+(.+)$/);
if (streetMatch) {
    simplified = streetMatch[1] + ' ' + streetMatch[2]; // "123 Main St"
}
```

### Enhanced behaviour
- If address matches a known `DATA.placeNicknames` entry → use nickname ("McDonald's", "St. Joe's")
- If numbered address → keep number + abbreviated street ("123 Main St")
- Result format: "123 Main St → McDonald's" or "St. Joe's → 45 Oak St"

### Apply after
Session 1 kernel data flow audit complete — do not apply before refactor.
